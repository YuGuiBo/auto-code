# 方案2：通用工具类架构说明

## 📋 概述

本项目采用**方案2：通用工具类架构**，旨在为多流程场景（请假、报销、采购等）提供可扩展的代码结构。

---

## 🏗️ 架构设计原则

1. **通用与业务分离**：通用工具类独立于具体业务，所有流程共用
2. **模块化组织**：每个流程（请假、报销等）独立成模块
3. **易于扩展**：新增流程只需复制 leave 模块结构
4. **API兼容**：保持向后兼容，测试脚本无需修改

---

## 🔧 核心组件说明

### 1. 通用工具类（common/util/）

#### ProcessMapper.java
**职责**：将 Flowable 的 ProcessInstance 转换为 ProcessInstanceDTO

**特点**：
- 只做基础字段映射（id、name、startTime 等）
- **不填充业务状态**（status 字段留给业务 Service 填充）
- 完全通用，任何流程都可以使用

**示例代码**：
```java
// 使用通用工具类
ProcessInstanceDTO dto = ProcessMapper.toDTO(processInstance);

// 业务Service自己填充业务状态
LeaveRequestStatus status = getLeaveStatus(processInstanceId);
dto.setStatus(status);
dto.setStatusDisplayName(status.getDisplayName());
```

#### TaskMapper.java
**职责**：将 Flowable 的 Task 转换为 TaskDTO

**特点**：
- 完全通用，与业务无关
- 静态方法，无需依赖注入
- 所有流程共用

---

### 2. 请假流程模块（leave/）

#### LeaveController.java
**API路径**：`/api/leave/*`

**主要接口**：
- `POST /api/leave/apply` - 申请请假
- `POST /api/leave/manager/approve/{taskId}` - 经理审批
- `POST /api/leave/dept-manager/approve/{taskId}` - 部门经理审批
- `GET /api/leave/process/instance/{id}` - 查询流程实例
- `GET /api/leave/task/list` - 查询待办任务

#### LeaveService.java
**职责**：
1. 请假业务逻辑（申请、审批）
2. 使用通用工具类进行数据转换
3. 自己管理业务状态（从流程变量的 `status` 字段读取）

**关键方法**：
```java
public ProcessInstanceDTO getProcessInstanceById(String processInstanceId) {
    // 1. 使用通用工具类转换
    ProcessInstanceDTO dto = ProcessMapper.toDTO(instance);
    
    // 2. 填充请假业务特有的状态
    LeaveRequestStatus status = getLeaveStatus(processInstanceId);
    dto.setStatus(status);
    dto.setStatusDisplayName(status.getDisplayName());
    
    return dto;
}

// 私有方法：获取请假状态（从流程变量读取）
private LeaveRequestStatus getLeaveStatus(String processInstanceId) {
    Map<String, Object> variables = runtimeService.getVariables(processInstanceId);
    String statusStr = (String) variables.get("status");
    return LeaveRequestStatus.valueOf(statusStr);
}
```

#### 2. 创建 ReimbursementService.java
```java
package com.example.flowable.reimbursement.service;

import com.example.flowable.reimbursement.enums.ReimbursementStatus;
import com.example.flowable.common.util.ProcessMapper;  // 使用通用工具类
import com.example.flowable.common.util.TaskMapper;     // 使用通用工具类
// ... 其他导入

@Service
public class ReimbursementService {
    
    @Autowired
    private RuntimeService runtimeService;
    
    @Autowired
    private TaskService taskService;
    
    @Autowired
    private HistoryService historyService;
    
    public ProcessInstanceDTO getProcessInstanceById(String processInstanceId) {
        ProcessInstance instance = runtimeService.createProcessInstanceQuery()
                .processInstanceId(processInstanceId)
                .singleResult();
        
        // 使用通用工具类转换
        ProcessInstanceDTO dto = ProcessMapper.toDTO(instance);
        
        // 填充报销业务特有的状态
        ReimbursementStatus status = getReimbursementStatus(processInstanceId);
        dto.setStatus(status);
        dto.setStatusDisplayName(status.getDisplayName());
        
        return dto;
    }
    
    private ReimbursementStatus getReimbursementStatus(String processInstanceId) {
        // 从流程变量读取报销状态
        Map<String, Object> variables = runtimeService.getVariables(processInstanceId);
        String statusStr = (String) variables.get("status");
        return ReimbursementStatus.valueOf(statusStr);
    }
}
```

## 🔍 关键设计决策

### 为什么 ProcessMapper 不填充业务状态？

**原因**：
1. **业务状态是特定于流程的**
   - 请假流程：PENDING_MANAGER_APPROVAL
   - 报销流程：PENDING_FINANCE_APPROVAL
   - 每个流程的状态枚举不同

2. **保持工具类的通用性**
   - ProcessMapper 完全通用，不依赖任何业务枚举
   - 可以用于任何流程

3. **职责清晰**
   - 工具类：负责字段映射
   - 业务Service：负责业务逻辑和状态管理
