# 📊 表结构设计说明

## 概述

本文档详细说明数据库存储与热加载方案中的表结构设计，以及与Flowable引擎表的关系。

---

## 📋 表结构总览

方案中创建了 **2张业务表**：

1. **process_definition** - 流程定义表（存储BPMN和YAML配置）
2. **process_deployment_history** - 流程部署历史表

---

## 📊 表1：process_definition（流程定义表）

### 表结构

这张表**同时存储BPMN和YAML配置**，不是分开的：

```sql
CREATE TABLE process_definition (
    id BIGSERIAL PRIMARY KEY,
    process_key VARCHAR(255) NOT NULL UNIQUE,
    process_name VARCHAR(255) NOT NULL,
    
    -- ⭐ 关键：BPMN和YAML都在这一张表里
    bpmn_content TEXT NOT NULL,      -- BPMN XML内容
    config_content TEXT NOT NULL,    -- YAML配置内容
    
    -- Flowable部署ID（记录引用，不是外键）
    deployment_id VARCHAR(255),
    deployed BOOLEAN DEFAULT FALSE,
    
    version INTEGER NOT NULL DEFAULT 1,
    api_prefix VARCHAR(100),
    description TEXT,
    enabled BOOLEAN DEFAULT TRUE,
    
    created_by VARCHAR(100),
    created_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_by VARCHAR(100),
    updated_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    deployed_time TIMESTAMP
);

-- 索引
CREATE INDEX idx_process_key ON process_definition(process_key);
CREATE INDEX idx_enabled ON process_definition(enabled);
CREATE INDEX idx_api_prefix ON process_definition(api_prefix);
```

### 字段说明

| 字段 | 类型 | 说明 | 重要性 |
|------|------|------|--------|
| **id** | BIGSERIAL | 主键 | ⭐⭐⭐ |
| **process_key** | VARCHAR(255) | 流程Key，对应BPMN中的process id | ⭐⭐⭐ |
| **process_name** | VARCHAR(255) | 流程名称（中文） | ⭐⭐⭐ |
| **bpmn_content** | TEXT | BPMN XML文件内容 | ⭐⭐⭐ |
| **config_content** | TEXT | YAML配置文件内容 | ⭐⭐⭐ |
| **deployment_id** | VARCHAR(255) | Flowable部署ID（记录引用） | ⭐⭐⭐ |
| **deployed** | BOOLEAN | 是否已部署到Flowable引擎 | ⭐⭐ |
| **version** | INTEGER | 版本号（自动递增） | ⭐⭐⭐ |
| api_prefix | VARCHAR(100) | API路径前缀 | ⭐⭐ |
| description | TEXT | 流程描述 | ⭐ |
| enabled | BOOLEAN | 是否启用 | ⭐⭐ |
| created_by | VARCHAR(100) | 创建人 | ⭐ |
| created_time | TIMESTAMP | 创建时间 | ⭐ |
| updated_by | VARCHAR(100) | 更新人 | ⭐ |
| updated_time | TIMESTAMP | 更新时间 | ⭐ |
| deployed_time | TIMESTAMP | 部署时间 | ⭐ |

### 职责

```
process_definition 表的职责：
├── 📝 存储原始内容（BPMN XML + YAML文本）
├── 📌 版本管理（version字段，支持升级）
├── 👤 审计信息（创建人、更新人、时间等）
├── 🚀 部署状态（deployed, deployment_id）
├── 🔄 支持回滚（保留历史版本）
└── 🔍 查询和管理（通过各种索引）
```

## 📊 表2：process_deployment_history（部署历史表）

### 表结构

记录每次部署的详细信息，支持审计和回滚：

```sql
CREATE TABLE process_deployment_history (
    id BIGSERIAL PRIMARY KEY,
    process_definition_id BIGINT NOT NULL,   -- 关联我们的业务表
    deployment_id VARCHAR(255),              -- Flowable部署ID
    version INTEGER NOT NULL,
    deployment_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    deployment_status VARCHAR(50),           -- SUCCESS, FAILED
    error_message TEXT,
    deployed_by VARCHAR(100),
    bpmn_snapshot TEXT,                      -- BPMN快照（用于回滚）
    config_snapshot TEXT,                    -- 配置快照（用于回滚）
    
    FOREIGN KEY (process_definition_id) REFERENCES process_definition(id)
);

-- 索引
CREATE INDEX idx_process_definition_id ON process_deployment_history(process_definition_id);
CREATE INDEX idx_deployment_time ON process_deployment_history(deployment_time);
```

### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| **id** | BIGSERIAL | 主键 |
| **process_definition_id** | BIGINT | 关联process_definition表的ID |
| **deployment_id** | VARCHAR(255) | Flowable部署ID |
| **version** | INTEGER | 部署时的版本号 |
| **deployment_time** | TIMESTAMP | 部署时间 |
| **deployment_status** | VARCHAR(50) | 部署状态（SUCCESS/FAILED） |
| **error_message** | TEXT | 部署失败时的错误信息 |
| **deployed_by** | VARCHAR(100) | 部署人 |
| **bpmn_snapshot** | TEXT | BPMN内容快照（用于回滚） |
| **config_snapshot** | TEXT | YAML配置快照（用于回滚） |

### 职责

```
process_deployment_history 表的职责：
├── 📜 记录每次部署的详细信息
├── ✅ 记录部署成功/失败状态
├── 🐛 保存错误信息（失败时）
├── 💾 保存内容快照（支持回滚）
├── 📊 提供部署审计追踪
└── 🔙 支持回滚到任意历史版本
```

## 🔗 与Flowable表的关系

### Flowable引擎表（既存的表）

Flowable有很多内部表，主要的有：

| 表名 | 说明 |
|------|------|
| **ACT_RE_DEPLOYMENT** | 部署表 |
| **ACT_RE_PROCDEF** | 流程定义表 |
| **ACT_GE_BYTEARRAY** | 二进制资源表（存储BPMN文件） |
| ACT_RU_EXECUTION | 流程实例执行表 |
| ACT_RU_TASK | 运行时任务表 |
| ACT_HI_PROCINST | 历史流程实例表 |

### 关联方式

**重要**：我们**不直接外键关联**Flowable的表，而是通过字段记录引用：

```
┌─────────────────────────────────────────────────────────────────┐
│                      我们的业务表                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │           process_definition                              │  │
│  │                                                           │  │

                                    通过deployment_id查询
                                    （不是外键关联！）
                                                    │
                                                    ↓
┌─────────────────────────────────────────────────│─────────────┐
│                    Flowable引擎表                 │              │
│  ┌──────────────────────────────────────────────│─────────┐  │
│  │        ACT_RE_DEPLOYMENT                      │         │  │
│  │                                                │         │  │
│  │  ID_: "abc123" <──────────────────────────────┘         │  │

│  │  DEPLOYMENT_ID_: "abc123"                                │  │
│  └──────────────────────────────────────────────────────────┘  │

│  ┌──────────────────────────────────────────────────────────┐  │
│  │        ACT_GE_BYTEARRAY                                   │  │
│  │                                                           │  │
│  │  ID_: "bytes123"                                          │  │
│  │  NAME_: "leaveRequestProcess.bpmn20.xml"                 │  │
│  │  BYTES_: [BPMN二进制内容]                                 │  │

│  └──────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### 为什么不直接外键关联？

**原因1：职责分离**
```
业务表（我们管理）          Flowable表（引擎管理）
      ↓                            ↓
  存储原始内容                 存储解析后数据
  版本管理                     运行时使用
  审计追踪                     自动管理
  支持回滚                     性能优化
```

**原因2：解耦设计**
- Flowable表是引擎内部表，结构可能随版本变化
- 外键约束会增加复杂度和维护成本
- 通过ID引用更灵活，解耦更彻底

**原因3：操作方式不同**
- 业务表：直接INSERT/UPDATE
- Flowable表：通过API操作（RepositoryService）

## 🎯 核心设计理念

### 设计原则

```
┌─────────────────────────────────────────────────────────────┐
│                    设计原则金字塔                            │
│                                                              │
│                         职责分离                              │
│                       ↗            ↖                         │
│                    原始内容          解析数据                 │
│                   (业务表)          (引擎表)                  │
│                  ↗        ↖      ↗        ↖                 │
│              版本管理    审计   运行时    性能                 │
│             ↗    ↖    ↗    ↖  ↗   ↖   ↗   ↖               │
│          保存  回滚  追踪  合规 执行 缓存 查询 优化            │
└─────────────────────────────────────────────────────────────┘
```

### 业务表 vs 引擎表

| 维度 | 业务表(process_definition) | Flowable引擎表 |
|------|---------------------------|----------------|
| **存储内容** | 📝 原始BPMN XML和YAML文本 | 🔧 解析后的结构化数据 |
| **主要用途** | 📦 版本管理、审计、回滚 | 🚀 运行时引擎使用 |
| **更新方式** | ✍️ 直接INSERT/UPDATE | 🔌 通过RepositoryService API |
| **生命周期** | 📅 长期保存所有历史版本 | ⚡ 引擎管理，可能清理旧版本 |
| **是否必需** | ⚠️ 可选（但强烈推荐） | ✅ 必需（引擎运行依赖） |
| **谁管理** | 👨‍💻 我们的应用程序 | 🤖 Flowable引擎 |

## 🤔 常见误解澄清

### 误解1："BPMN和配置分两张表存储"

❌ **错误理解**：
```
process_bpmn 表 ← 存储BPMN
      ↓
process_config 表 ← 存储YAML
      ↓
通过外键关联
```

✅ **正确理解**：
```
process_definition 表
├── bpmn_content    ← BPMN XML
└── config_content  ← YAML配置
（都在同一张表的不同字段）
```

**原因**：
- BPMN和YAML是一一对应的，分表没有意义
- 一起存储方便版本管理和原子操作
- 简化数据结构和查询逻辑

---

### 误解2："通过外键关联Flowable表"

❌ **错误理解**：
```sql
CREATE TABLE process_definition (
    deployment_id VARCHAR(255),
    FOREIGN KEY (deployment_id) 
        REFERENCES act_re_deployment(id_)  -- ❌ 错误！
);
```

✅ **正确理解**：
```sql
CREATE TABLE process_definition (
    deployment_id VARCHAR(255),  -- 只是记录ID，不是外键
    -- 没有 FOREIGN KEY 约束
);
```

**原因**：
- Flowable表由引擎管理，不应该直接操作
- 外键约束会增加耦合和复杂度
- 通过ID引用更灵活，便于解耦

---

### 误解3："可以直接插入Flowable表来部署流程"

❌ **错误做法**：
```sql
-- ❌ 千万不要这样做！
INSERT INTO act_re_deployment (...);
INSERT INTO act_ge_bytearray (...);
INSERT INTO act_re_procdef (...);
```

✅ **正确做法**：
```java
// ✅ 必须通过API
Deployment deployment = repositoryService.createDeployment()
    .addString("process.bpmn20.xml", bpmnContent)
    .deploy();
```

**原因**：
- 引擎会做复杂的解析、验证、缓存
- 直接插表会导致缓存不一致
- 版本管理、关联关系等由引擎自动处理

---

### 误解4："业务表是可选的"

⚠️ **需要澄清**：

从技术角度：
- 只要调用API部署，Flowable就能正常运行
- 业务表在技术上是"可选的"

从实践角度：
- ✅ **强烈推荐**创建业务表
- 原因：
  - 支持版本管理和回滚
  - 提供审计追踪
  - 便于运维管理
  - 支持热加载和动态更新

## ✅ 设计优势

### 1. 职责清晰

```
业务层面（业务表）          技术层面（引擎表）
      ↓                           ↓
  版本管理                    流程执行
  审计追踪                    性能优化
  数据备份                    缓存管理
  回滚支持                    自动维护
```

### 2. 解耦设计

- 业务逻辑和引擎实现分离
- 便于系统升级和维护
- 支持多种部署方式

### 3. 灵活扩展

- 可以添加自定义字段
- 支持多种业务需求
- 便于集成其他系统

### 4. 运维友好

- 清晰的部署历史
- 便于问题排查
- 支持快速回滚

## 📝 总结

### 核心要点

1. **2张业务表**：
   - `process_definition` - 同时存BPMN和YAML
   - `process_deployment_history` - 部署历史

2. **不是外键关联**：
   - 通过 `deployment_id` 字段记录引用
   - 不直接操作Flowable表

3. **必须通过API**：
   - 使用 `RepositoryService.createDeployment()`
   - 不能直接插入引擎表

4. **职责分离**：
   - 业务表：原始内容、版本管理、审计
   - 引擎表：解析数据、运行时使用

### 实施建议

✅ **推荐做法**：
1. 创建业务表存储原始内容
2. 通过API部署到Flowable
3. 记录部署历史支持回滚
4. 实现热加载机制

❌ **避免做法**：
1. 直接操作Flowable引擎表
2. 使用外键关联引擎表
3. 绕过API直接插入数据
4. 忽略版本管理和审计
