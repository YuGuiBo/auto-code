# 🎯 Flowable流程定义数据库存储与热加载实施方案

## 📋 方案概述

将当前基于静态文件的流程定义（BPMN和YAML配置）完全迁移到数据库存储，实现热加载机制，使得前端可以动态上传流程定义并立即生效。

### 核心目标

1. ✅ **数据库存储**：BPMN文件和YAML配置完全存储在PostgreSQL数据库
2. ✅ **热部署**：上传后立即调用Flowable API完成部署，流程立即生效
3. ✅ **热加载**：配置更新后自动重新加载，无需重启应用
4. ✅ **版本管理**：支持流程定义的版本控制和历史记录
5. ✅ **向后兼容**：保持现有API接口不变，测试脚本无需修改

---

## 🏗️ 架构设计

### 核心改造思路

```
前端上传 → 存入业务表 → 调用Flowable部署API → 立即生效
              ↓
    程序重启 → 从数据库加载 → 批量部署
```

### 关键设计原则

1. **两步存储法**：
   - 第一步：保存到业务表 `process_definition`（存储原始内容）
   - 第二步：调用 `RepositoryService.createDeployment()` 部署到Flowable引擎

2. **为什么不能直接插入Flowable引擎表？**
   - ❌ 复杂的关联关系：`ACT_RE_PROCDEF`、`ACT_GE_BYTEARRAY`、`ACT_RE_DEPLOYMENT`等表之间有复杂外键
   - ❌ 缓存管理：引擎维护内存缓存，直接插表会导致缓存不一致
   - ❌ 版本管理：引擎会自动管理版本号，手动插入容易出错
   - ❌ 校验逻辑：部署过程会进行XML校验、节点校验等

3. **业务表与引擎表的关系**：
   ```
   业务表（process_definition）        Flowable引擎表
           ↓                                ↓
   存储BPMN和YAML原始内容          存储解析后的流程定义
           ↓                                ↓
         保存                          调用部署API
           ↓                                ↓
      版本管理、审计                 运行时引擎使用
   ```

## 🗄️ 数据库设计

### 表1：流程定义主表 `process_definition`

```sql
CREATE TABLE process_definition (
    id BIGSERIAL PRIMARY KEY,
    process_key VARCHAR(255) NOT NULL UNIQUE,           -- 流程Key（如：leaveRequestProcess）
    process_name VARCHAR(255) NOT NULL,                  -- 流程名称
    version INTEGER NOT NULL DEFAULT 1,                  -- 版本号
    bpmn_content TEXT NOT NULL,                          -- BPMN文件内容（XML）
    config_content TEXT NOT NULL,                        -- YAML配置内容
    api_prefix VARCHAR(100),                             -- API路径前缀
    description TEXT,                                    -- 流程描述
    deployment_id VARCHAR(255),                          -- Flowable部署ID
    deployed BOOLEAN DEFAULT FALSE,                      -- 是否已部署
    enabled BOOLEAN DEFAULT TRUE,                        -- 是否启用
    created_by VARCHAR(100),                             
    created_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_by VARCHAR(100),
    updated_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    deployed_time TIMESTAMP
);

-- 创建索引
CREATE INDEX idx_process_key ON process_definition(process_key);
CREATE INDEX idx_enabled ON process_definition(enabled);
CREATE INDEX idx_api_prefix ON process_definition(api_prefix);

-- 添加注释
COMMENT ON TABLE process_definition IS '流程定义表：存储BPMN和YAML配置';
```

### 表2：流程部署历史表 `process_deployment_history`

```sql
CREATE TABLE process_deployment_history (
    id BIGSERIAL PRIMARY KEY,
    process_definition_id BIGINT NOT NULL,
    deployment_id VARCHAR(255),
    version INTEGER NOT NULL,
    deployment_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    deployment_status VARCHAR(50),                       -- SUCCESS, FAILED
    error_message TEXT,
    deployed_by VARCHAR(100),
    bpmn_snapshot TEXT,                                  -- BPMN快照（用于回滚）
    config_snapshot TEXT,                                -- 配置快照
    FOREIGN KEY (process_definition_id) REFERENCES process_definition(id)
);

CREATE INDEX idx_process_definition_id ON process_deployment_history(process_definition_id);
CREATE INDEX idx_deployment_time ON process_deployment_history(deployment_time);
```

private void deployToFlowable(ProcessDefinitionEntity definition, String deployedBy) {
    // 使用RepositoryService完成部署

    definition.setDeploymentId(deployment.getId());
    definition.setDeployed(true);
    processRepository.save(definition);
    
    // 记录部署历史
    recordDeploymentHistory(definition, deployedBy, "SUCCESS", null);
}

#### 5. DatabaseProcessConfigLoader（核心改造）
- **职责**：从数据库加载配置并部署到Flowable
- **关键方法**：

```java
@Component
public class DatabaseProcessConfigLoader implements ApplicationRunner {
    
    @Autowired
    private ProcessDefinitionRepository processRepository;
    
    @Autowired
    private RepositoryService repositoryService;
    
    // 应用启动时自动执行
    @Override
    public void run(ApplicationArguments args) throws Exception {
        loadAllConfigsFromDatabase();
    }
    
    // 从数据库加载所有流程配置并部署
    public synchronized void loadAllConfigsFromDatabase() {
        // 1. 查询所有启用的流程定义
        List<ProcessDefinitionEntity> definitions = processRepository.findByEnabledTrue();
        
        for (ProcessDefinitionEntity definition : definitions) {
            // 2. 解析YAML配置
            ProcessConfig config = yamlMapper.readValue(
                definition.getConfigContent(), ProcessConfig.class);
            
            // 3. 缓存配置到内存
            processConfigs.put(definition.getProcessKey(), config);
            
            // 4. 如果未部署，则部署到Flowable
            if (!definition.getDeployed()) {
                deployToFlowable(definition);
            }
        }
    }
    
    // 热加载：重新加载所有配置
    public synchronized void reloadConfigs() {
        processConfigs.clear();
        loadAllConfigsFromDatabase();
    }
}
```

## 🔄 数据迁移方案

### ProcessDataMigration（启动时自动迁移）

```java
@Component
public class ProcessDataMigration implements ApplicationRunner {
    
    @Autowired
    private ProcessDefinitionRepository processRepository;
    
    @Override
    public void run(ApplicationArguments args) throws Exception {
        // 检查数据库是否为空，如果为空则迁移
        if (processRepository.count() == 0) {
            migrateFromFiles();
        }
    }
    
    private void migrateFromFiles() throws Exception {
        // 1. 扫描 src/main/resources/processes/ 目录
        Resource[] bpmnResources = resolver.getResources(
            "classpath:processes/**/*.bpmn20.xml");
        
        for (Resource bpmnResource : bpmnResources) {
            // 2. 读取BPMN文件内容
            String bpmnContent = new String(
                bpmnResource.getInputStream().readAllBytes());
            
            // 3. 查找对应的YAML配置文件
            String configPath = bpmnPath.replace(".bpmn20.xml", "-config.yml");
            String configContent = new String(
                configResource.getInputStream().readAllBytes());
            
            // 4. 解析配置获取processKey
            ProcessConfig config = yamlMapper.readValue(
                configContent, ProcessConfig.class);
            
            // 5. 保存到数据库
            ProcessDefinitionEntity entity = new ProcessDefinitionEntity();
            entity.setProcessKey(config.getProcess().getKey());
            entity.setBpmnContent(bpmnContent);
            entity.setConfigContent(configContent);
            // ... 设置其他字段
            
            processRepository.save(entity);
        }
    }
}
```

**注意**：迁移完成后，可以在 `application.yml` 中设置 `app.process.migration.enabled=false` 来禁用自动迁移。

## 🔍 关键问题解答

### Q1: 为什么必须调用 RepositoryService.createDeployment()?

**答：** 
- 仅仅把BPMN存入数据库是无效的
- 必须调用Flowable的部署API，引擎才会：
  - 解析XML并验证格式
  - 将流程定义加载到内存缓存
  - 写入引擎内部表（ACT_RE_PROCDEF等）
  - 自动管理版本号
- 直接插入引擎表会导致缓存不一致和运行时错误

### Q2: 数据库表和Flowable引擎表的区别？

| 对比项 | 业务表(process_definition) | Flowable引擎表(ACT_*) |
|-------|---------------------------|----------------------|
| **存储内容** | 原始BPMN和YAML文本 | 解析后的结构化数据 |
| **主要用途** | 版本管理、审计、回滚 | 运行时引擎使用 |
| **更新方式** | 直接INSERT/UPDATE | 通过RepositoryService |
| **是否必需** | ❌ 可选（但推荐） | ✅ 必需 |

### Q3: 如何处理流程升级？

**答：**
1. 上传新版本时，业务表的 `version` 字段自动递增
2. Flowable引擎会自动创建新版本的流程定义
3. 旧的流程实例继续使用旧版本
4. 新的流程实例使用新版本
5. 可以通过部署历史表回滚到任何版本

### Q4: 热加载如何实现？

**答：**
1. **上传时自动触发**：`uploadProcessDefinition()` 方法最后调用 `configLoader.loadAllConfigsFromDatabase()`
2. **手动触发**：调用 `/api/admin/process-definition/reload` 接口
3. **定时检查**（可选）：使用 `@Scheduled` 定期检查数据库更新
4. **消息通知**（可选）：使用Redis Pub/Sub通知其他实例

## ⚠️ 注意事项

### 1. 事务管理
- 数据库保存和Flowable部署必须在同一事务内
- 使用 `@Transactional` 注解确保原子性
- 部署失败时事务会回滚，数据库记录不会保存

### 2. 并发控制
- 使用数据库悲观锁防止并发部署：`@Lock(LockModeType.PESSIMISTIC_WRITE)`
- `loadAllConfigsFromDatabase()` 方法使用 `synchronized` 防止并发加载

### 3. 性能优化
- 启动时批量加载，避免逐个查询
- 考虑使用 `@Cacheable` 缓存配置对象
- 定时同步时使用 `updated_time` 索引

### 4. 错误处理
- 所有部署错误都要记录到历史表
- BPMN和YAML验证要有详细的错误信息
- API接口要返回友好的错误提示

## 🎯 总结

### 核心要点

1. **两步存储**：业务表存原始内容 + Flowable引擎表存解析结果
2. **必须部署**：调用 `RepositoryService.createDeployment()` 是必需的
3. **热加载**：上传后自动触发 `loadAllConfigsFromDatabase()`
4. **版本管理**：业务表和Flowable引擎都支持版本控制
